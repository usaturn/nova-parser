# Claude Code `feature-dev` 詳解ガイド

`feature-dev` は、Claude Code で新機能開発を進めるためのプラグインです。単に「コードを書かせる」のではなく、要件整理、既存コードの調査、設計比較、実装、レビュー、完了サマリまでを 1 本のワークフローにまとめているのが特徴です。

2026-03-29 時点で確認した一次情報では、`feature-dev` は Anthropic 提供のプラグインとして公開されており、プラグインページでは Claude Code 向けに `/feature-dev` を実行して使うものとして案内されています。

## 概要

| 項目 | 内容 |
|---|---|
| プラグイン名 | `feature-dev` |
| バージョン | `1.0.0` |
| 作者 | Sid Bidasaria |
| 主コマンド | `/feature-dev` |
| 主な補助エージェント | `code-explorer`, `code-architect`, `code-reviewer` |
| 補助エージェントのモデル | 3 つとも `sonnet` |
| 主な目的 | 新機能開発を、調査・設計・レビュー込みで体系的に進めること |

このプラグインの価値は、実装前に必ず次の順番を踏ませる点にあります。

- まず要求を曖昧なまま進めない
- 次に既存コードの構造を調べる
- そのうえで設計案を比較する
- ユーザ承認後にだけ実装する
- 実装後は別視点のレビューをかける

つまり、`feature-dev` は「実装を速くするコマンド」というより、「雑な実装に流れないための開発手順を Claude に強制するコマンド」です。

## `feature-dev` がやっていること

プラグイン本体の `commands/feature-dev.md` を見ると、このコマンドは単純なテンプレートではなく、かなり明確な行動規則を Claude に与えています。特に重要なのは次の 5 点です。

- 曖昧点があれば必ず具体的に質問する
- 既存コードを理解する前に設計や実装へ進まない
- 補助エージェントが挙げた重要ファイルを本体も読み直す
- 進捗は Todo で追跡する
- 実装はユーザの明示承認が出るまで始めない

このため、普段の「まず書いてみる」型のやり方よりも遅く感じることがあります。ただし、その分だけ設計ミス、既存規約との不整合、レビュー漏れを減らしやすくなります。

## 7 フェーズの詳細

### 1. Discovery

目的は「何を作るのか」を固定することです。

Claude は最初に feature request を見て、曖昧なら次を聞きます。

- 何の問題を解決したいのか
- その機能に何をしてほしいのか
- 制約や要求はあるか

そして理解内容を要約し、ユーザに確認します。ここで要求の土台を固めないと、後続フェーズすべてがずれます。

### 2. Codebase Exploration

目的は「このコードベースではどう作るべきか」を知ることです。

ここでは 2 から 3 個の `code-explorer` エージェントを並列起動し、たとえば次の観点を分担して調べさせます。

- 似た機能が既にあるか
- 対象領域のアーキテクチャはどうなっているか
- UI、API、テスト、拡張ポイントはどう実装されているか

重要なのは、エージェントに調査させて終わりではない点です。コマンド定義では、エージェントが返した重要ファイルを本体の Claude も読み、最終的に総合理解を作るよう指示されています。つまり `feature-dev` は、サブエージェントの報告をそのまま鵜呑みにせず、メインスレッドでも読み直す設計です。

### 3. Clarifying Questions

目的は、設計前に曖昧さをゼロに近づけることです。

このフェーズはコマンド定義でも `CRITICAL` と明記されており、スキップ禁止です。Claude は調査結果と元要求を見て、次のような未確定要素を洗い出します。

- エッジケース
- エラー処理
- 既存機能との統合点
- スコープ境界
- 後方互換性
- 性能要求
- UI/UX の好み

質問は整理された一覧として提示され、回答が返るまで次の設計フェーズへ進まないようになっています。ここが `feature-dev` の強い部分で、実装前の確認不足をかなり防げます。

### 4. Architecture Design

目的は「どの実装方針で進めるか」を比較し、意思決定することです。

ここでは 2 から 3 個の `code-architect` エージェントを並列起動し、たとえば次の軸で案を作らせます。

- 最小変更で済ませる案
- 保守性を優先する案
- 速度と品質のバランス案

README ではこのフェーズを「複数案比較」として説明しています。一方で `code-architect` 自体のプロンプトは「各エージェントは自分の案を断定的に設計する」方向です。つまり、`feature-dev` 全体としては複数案を比べますが、各設計エージェントの中では優柔不断にせず、はっきり選んだ設計図を出す構成になっています。

ここで Claude は次をユーザに提示します。

- 各案の要約
- トレードオフ
- 自分の推奨案
- 実際にどのファイルや責務が変わるか

その後、どの案で進めるかをユーザに確認します。

### 5. Implementation

目的は実装です。

ただし、このフェーズには `DO NOT START WITHOUT USER APPROVAL` が明記されています。つまり、`feature-dev` は設計まで終わっても、承認なしに勝手に書き始めない前提です。

実装フェーズでは次を守ります。

- 前段で特定した関連ファイルを読む
- 選ばれたアーキテクチャに従う
- 既存のコード規約に合わせる
- Todo を更新しながら進める

このため、ただの一発生成ではなく、「選んだ設計の実装フェーズ」に移る感覚で使うのが正しいです。

### 6. Quality Review

目的は、書いたコードが動くだけでなく、壊れにくく読みやすいかを確認することです。

ここでは 3 個の `code-reviewer` エージェントを並列起動し、少なくとも次の視点でレビューします。

- シンプルさ、DRY、可読性
- バグやロジックの誤り
- プロジェクト規約や抽象化の妥当性

`code-reviewer` の定義で特に重要なのは、レビュー対象が原則 `git diff` であることと、信頼度 80 以上の指摘しか報告しないことです。つまり、ノイズの多い感想ではなく、「かなり確度の高い問題」に絞る設計です。

レビュー後、Claude は findings をまとめ、ユーザに次の判断を求めます。

- 今すぐ直す
- 後で直す
- そのまま進める

この「レビュー結果に対する意思決定」を挟むのも `feature-dev` の特徴です。

### 7. Summary

目的は、変更内容を最後に明文化することです。

最後に Claude は次を要約します。

- 何を作ったか
- 重要な設計判断
- 変更したファイル
- 次にやるとよいこと

実装だけで終わらず、変更の説明責任まで含めて締める設計になっています。

## 3 つの補助エージェント

### `code-explorer`

既存機能の実装を深く追う調査役です。エントリポイント、コールチェーン、データ変換、依存関係、アーキテクチャ層まで追跡し、重要ファイル一覧も返します。

要するに、「似た実装を雑に grep する役」ではなく、「どう動いているかを入口から出口までたどる役」です。

### `code-architect`

既存パターンと規約を踏まえて、新機能の実装設計図を作る役です。ファイル単位の変更計画、責務分割、データフロー、実装順序まで落とし込みます。

単に「こういう考え方もあります」と述べるのではなく、かなり具体的な設計図を出す前提になっています。

### `code-reviewer`

実装後のレビュー役です。`CLAUDE.md` 等のプロジェクトルール、実バグ、品質問題を確認し、高信頼な問題だけを挙げます。

このエージェントの設計思想は「大量の指摘を出すこと」ではなく「本当に重要な指摘だけに絞ること」です。

## `feature-dev` が向いているケース

- 複数ファイルにまたがる機能追加
- 既存設計への自然な乗せ方を考えたい変更
- 要件が少し曖昧で、先に質問して詰めたい変更
- 実装前に設計案の比較をしたい変更
- 実装後に自動レビューも含めて進めたい変更

## 向かないケース

README でも、次のようなものには不向きとされています。

- 1 行レベルの修正
- 自明な単純変更
- 仕様が完全に確定した小タスク
- 急ぎの hotfix

理由は単純で、`feature-dev` は慎重な手順を踏むぶん、軽作業にはオーバーヘッドが大きいからです。

## 使うときのコツ

- 最初の依頼文に、目的、制約、非機能要件を書く
- 既存仕様を壊したくない箇所は最初に明示する
- フェーズ 3 の質問に雑に答えない
- フェーズ 4 では「最小変更」か「保守性重視」かを意識して選ぶ
- フェーズ 6 のレビューは省略せず、少なくとも一度は見てから判断する

## 簡単な使い方

### 前提

- Claude Code がインストール済みであること
- 対象が Git リポジトリであること
- 既存コードベースがあること
- 単発の小修正ではなく、ある程度まとまった機能追加であること

### 導入

一番簡単なのは、プラグインページから Claude Code にインストールする方法です。

- `https://claude.com/plugins/feature-dev` を開く
- `Install in Claude Code` を使う

CLI やプラグインマネージャ経由で入れる場合は、Claude Code の公式ドキュメントにあるプラグイン機能を使います。Anthropic のデモ marketplace として `anthropics/claude-code` を追加し、`/plugin` の Discover タブから `feature-dev` を選んで入れるのが確実です。

```text
/plugin marketplace add anthropics/claude-code
/plugin
```

インストール後は、必要に応じてプラグイン再読込を行います。

```text
/reload-plugins
```

### 最小手順

1. Claude Code を対象リポジトリで開く
2. `feature-dev` をインストールする
3. `/feature-dev` に、作りたい機能を続けて書く
4. 途中の質問に答える
5. 提案された設計案から 1 つ選ぶ
6. 実装開始の確認に明示的に同意する
7. レビュー結果に対して「今直すか」を決める

### 実行例

```text
/feature-dev 管理画面に CSV エクスポート機能を追加したい
```

より具体的に書くならこちらです。

```text
/feature-dev 管理画面に CSV エクスポート機能を追加したい。対象は検索結果一覧。UTF-8 with BOM で出力し、権限は管理者のみ。既存の一覧フィルタ条件をそのまま反映したい
```

このように書くと、フェーズ 1 と 3 の質問が減り、探索と設計に早く入れます。

### 実際の流れ

`/feature-dev` を実行すると、概ね次の順に進みます。

1. 何を作るかを確認される
2. 既存コードの類似機能や構造を調べる
3. 足りない仕様について質問される
4. 実装案が複数提示される
5. 選んだ案でよいか確認される
6. 実装開始の承認を求められる
7. 実装後にレビュー結果が出る
8. 何を変更したかが最後に要約される

## まとめ

`feature-dev` は、Claude Code に「新機能開発の基本手順」を守らせるためのプラグインです。特に強いのは、実装前のコード理解、設計比較、実装後レビューを 1 本の流れにしている点です。

逆に言うと、即席でコードを吐かせたいときには重いです。小修正には向きませんが、複数ファイルへ触る機能追加や、既存コードへの自然な組み込みを重視したい場面ではかなり相性がよいです。
