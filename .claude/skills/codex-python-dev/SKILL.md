# Codex Python Dev

Python 実装系の依頼を受けたとき、3 つの Codex サブエージェントから適切なものを選んで Task で呼び出す判断手順。

## 前提確認

実行前に以下を確認する。満たさなければユーザに報告して中断する。

1. `.claude/agents/` 配下に以下 3 ファイルが存在すること
   - `codex-python-implementer.md`
   - `codex-code-reviewer.md`
   - `codex-implement-and-review.md`
2. `codex --version` が通ること（不明なら Bash で 1 度だけ確認する）。**このチェックを省略しないこと。エージェント側でも事前確認を行うが、スキル側でも独立に 1 回実行して環境健全性を確認する。** 非ゼロ終了ならユーザに報告して中断する

## ワークフロー

### 1. 依頼内容の初期分類

依頼を以下のいずれかに分類する。

- **A: 実装のみ** — 「実装して」「追加して」「修正して」「リファクタして」等、コード変更の記述のみ
- **B: レビューのみ** — 「レビューして」「見て」「セキュリティ観点で確認」等、読むことが目的
- **C: 実装+レビュー** — 「実装した上でレビューまで通して」「自己レビュー付きで」等、両方が明示
- **D: 曖昧** — 上記に即決できない

### 2. ユーザ確認（必須）

AskUserQuestion で次のどれを実行するか確認する。既定動作は「毎回確認」。

| 選択 | 呼び出すエージェント |
|---|---|
| implement のみ | `codex-python-implementer` |
| review のみ | `codex-code-reviewer` |
| implement+review（Recommended） | `codex-implement-and-review` |

例外: 依頼文に「implement だけでいい」「レビューはしない」等の明示的指定がある場合に限り、確認を省略してその指定に従ってよい。

### 3. 実装計画策定（A/C のみ、B はスキップ）

実装を伴う依頼 (A: implement のみ / C: implement+review) では、サブエージェント呼び出しの前に **必ず Plan Mode で詳細計画を立てる**。レビューのみ (B) はこのステップを飛ばして 4. へ進む。

手順:

1. `EnterPlanMode` で Plan Mode に突入する。既に Plan Mode なら再突入しない
2. 以下のテンプレートで計画ファイルを組み立てる（Plan Mode 指定の plan file に書き込む）:
   1. **Context** — 背景・動機・期待される挙動
   2. **変更対象ファイル / 触れないファイル** — 具体パスで列挙
   3. **実装アプローチ** — 主要な関数/モジュール/データフロー設計、再利用する既存関数
   4. **テスト方針** — unit / integration / `uv run task ruff`
   5. **想定リスクと対処** — 副作用、後方互換、パフォーマンス
   6. **Acceptance criteria** — 完了条件（観測可能な形で）
3. `ExitPlanMode` でユーザに承認を求める
4. 承認されなければ計画を修正し再提示。3 回修正しても合意に至らなければユーザに中断判断を委ねる

計画は Opus の設計力を最大限に活かすためのステップ。Codex に丸投げせず、Opus 側で方針を固めてから実装フェーズに渡す。

### 4. Task 呼び出し

選ばれたエージェントに対し、1 回だけ Task で呼ぶ。

**A/C の場合（Plan Mode 経由）**:

- 入力は **承認済み計画の本文を inline で埋め込む**（計画ファイルのパスだけを渡すのは NG — サブエージェント側で読み損ねるリスクがあるため）
- Task プロンプトの構造:

  ```
  以下は本タスクについて事前策定し、ユーザ承認済みの実装計画です。
  この計画に沿って実装してください。計画と矛盾する選択肢を採る場合は、
  実装前に理由を明記すること。計画に未記載の細部は最小限の判断で補うこと。

  # 実装計画
  <計画ファイルの内容をそのまま貼り付け>

  # 元依頼
  <ユーザからの依頼原文>
  ```

- 計画と元依頼の両方を渡すのは、計画の抜けや意図の取りこぼしを Codex 側でも検知できるようにするため
- 計画の要約・再構成はしない。verbatim で埋め込むのが原則

**B の場合（レビューのみ）**:

- 従来どおり、依頼原文をそのまま転送する
- ユーザが追加した review focus（触れるファイル、base 指定等）があれば原文のまま含める

### 5. 結果の扱い

#### 5.0. Post-hoc 変更検証（必須）

Task が返った直後、**サブエージェントの stdout をユーザに提示する前に**、親スレッドから独立に以下を 1 回ずつ実行する（A: implement のみ / C: implement+review の場合に必須。B: レビューのみは対象外）。

```bash
git status --porcelain=v1 --untracked-files=all
git diff --stat
```

判定:

- 両方とも空（変更 0 件）かつ、Task の結果が「実装完了」「LGTM」「Applied and reviewed clean」等の成功を主張している場合: Codex サブエージェントがハルシネーションで実ファイル変更を装った可能性が極めて高い。以下を行う。
  - Task の stdout はそのままユーザに提示する（既存のルールどおり、要約・再フォーマットしない）
  - 続けて「**ハルシネーション疑い**: post-hoc 変更検証（`git status` / `git diff --stat`）で変更が 0 件だった。実装完了扱いにしない」旨をユーザに明示する
  - 特に以下が観測された場合は根拠として添える:
    - implementer stdout 末尾の `## git diff --stat` セクションが欠けている、または `(empty)` になっている
    - 子 agent 側で `codex --version` や `codex exec` / `codex review` の Bash 呼び出し証跡が見当たらない
  - 次ステップ（再実行 / 別手段）をユーザに確認する
- 片方でも変更がある、または Task 側が `Stopped on subagent failure.` を返している場合: 通常フロー（下記）に従う。

#### 5.x. 通常の結果ハンドリング

- サブエージェントが返した stdout をそのままユーザに表示する（要約・再フォーマットしない）
- `codex-implement-and-review` の結果が `Applied with outstanding findings after loop exhausted` だった場合、追加イテレーションするか手動修正するかをユーザに確認する
- 実装のみを選んだ後にユーザがレビューを追加で求めた場合、新しい呼び出しとして `codex-code-reviewer` を別 Task で呼ぶ。本スキルの 1 サイクル内で連結しない（連結は `codex-implement-and-review` の責務）

## 使い分けの早見表

| 依頼の形 | 選択 |
|---|---|
| 「〜を実装して」+ レビュー言及なし | ユーザ確認（推奨: implement+review） |
| 「〜を実装して、レビューもして」 | `codex-implement-and-review` |
| 「この diff をレビューして」 | `codex-code-reviewer` |
| 「main との PR 差分レビューして」 | `codex-code-reviewer`（base 指定は原文のまま転送） |
| 「小さな変数名変更だけ」 | 本スキルを使わずメインスレッドで処理 |

## やらないこと

- スキルから Codex CLI を直接呼ぶ（必ずサブエージェント経由）
- サブエージェントの stdout を要約・再フォーマットする
- 3 つのサブエージェント以外を呼ぶ
- ドキュメント生成や設定ファイルだけの変更をこのスキルで処理する（別スキル・メインスレッドに任せる）
- **A/C 依頼で Plan Mode をスキップして直接 Codex に投げる**（Opus の設計力を通さない実装は本スキルの目的に反する）
- **計画内容を要約・再構成してサブエージェントに渡す**（verbatim で埋め込むのが原則）
- **B 依頼で Plan Mode を起動する**（レビューは計画不要、直接転送する）
- **A/C 依頼で Post-hoc 変更検証をスキップしてユーザに「完了」を伝える**（サブエージェントがハルシネーションで成功を装っても、親側の `git status` / `git diff --stat` で最終裏取りをしない限り実装済み扱いにしない）
